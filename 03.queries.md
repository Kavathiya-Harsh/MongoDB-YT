# MongoDB — Advanced Queries

> **Topic:** $not, $nor, $regex, $expr  
> **Level:** Intermediate  
> **Prerequisites:** Basic MongoDB — insert, find, comparison & logical operators

## 1. `$not` — Negate a Condition

Use `$not` when you want documents that **do NOT match** a condition.

### Syntax
```js
{ field: { $not: { <operator expression> } } }
```

### Example — Jobs needing 2 or fewer years of experience
```js
db.jobs.find({
  experience: { $not: { $gt: 2 } }
})
// Returns: Software Engineer (Startup)
```

### ⚠️ Common Mistake
```js
// ❌ WRONG — $not outside the field
{ $not: { experience: { $gt: 2 } } }

// ✅ CORRECT — $not inside the field
{ experience: { $not: { $gt: 2 } } }
```

---

## 2. `$nor` — None of These

Use `$nor` when you want documents that **match none** of the given conditions.  
Think of it as: `NOT (condition1 OR condition2)`

### Syntax
```js
{ $nor: [ { condition1 }, { condition2 }, ... ] }
```

### Example — Jobs that are neither from Amazon nor in Bangalore
```js
db.jobs.find({
  $nor: [
    { company: "Amazon" },
    { location: "Bangalore" }
  ]
})
// Returns: Software Engineer (Startup, Remote)
// Excluded: Flipkart (Bangalore), Swiggy (Bangalore), Amazon
```

> 💡 Great for building **exclusion filters** in search and filter APIs.

## 3. `$regex` — Pattern Matching on Strings

Use `$regex` to search string fields using regular expressions.

### Syntax
```js
{ field: { $regex: "pattern", $options: "flags" } }
```

### Common Patterns

| Pattern | Meaning | Example |
|---|---|---|
| `^Senior` | Starts with "Senior" | `"^Senior"` |
| `Developer$` | Ends with "Developer" | `"Developer$"` |
| `backend` + `i` flag | Case-insensitive match | `$options: "i"` |

### Examples
```js
// Roles starting with "Senior"
db.jobs.find({ role: { $regex: "^Senior" } })

// Roles ending with "Developer"
db.jobs.find({ role: { $regex: "Developer$" } })

// Case-insensitive search for "backend"
db.jobs.find({ role: { $regex: "backend", $options: "i" } })
```

### ⚠️ Performance Warning
- Regex **scans every document** — it does NOT use indexes properly
- Exception: patterns starting with `^` can use indexes
- **Avoid in high-traffic APIs** unless absolutely necessary

## 4. `$expr` — Compare Two Fields in the Same Document

Use `$expr` when your condition involves **comparing one field against another field** inside the same document. Regular operators can only compare a field to a fixed value.

### Syntax
```js
{ $expr: { <aggregation expression> } }
```

### Example — Jobs where salary > experience × 5,00,000
```js
db.jobs.find({
  $expr: {
    $gt: ["$salary", { $multiply: ["$experience", 500000] }]
  }
})
```

### Example — Jobs where experience < salary ÷ 10,00,000
```js
db.jobs.find({
  $expr: {
    $lt: ["$experience", { $divide: ["$salary", 1000000] }]
  }
})
```

> 💡 `$expr` uses **aggregation operators** like `$multiply`, `$divide`, `$add`, `$subtract`.  
> Fields are referenced with a `$` prefix — e.g., `"$salary"`, `"$experience"`.

### ⚠️ Performance Warning
- `$expr` runs **per document** — slower than regular queries
- Use only when fixed-value comparisons are not enough

## Combined Query Example

Real-world filter — mid-level Backend jobs in Bangalore with good salary:

```js
db.jobs.find({
  role:       { $regex: "Backend", $options: "i" },
  experience: { $gte: 3, $lte: 5 },
  location:   "Bangalore",
  salary:     { $gt: 1000000 }
})
```

This uses `$regex`, range filter, exact match, and numeric comparison — all in one query.

---

## Quick Summary

| Operator | Use When | Watch Out For |
|---|---|---|
| `$not` | Negate a field-level condition | Must be inside the field, not outside |
| `$nor` | Exclude multiple conditions | Excludes ALL matching docs — think carefully |
| `$regex` | Pattern match on strings | Slow — avoid in high-traffic APIs |
| `$expr` | Compare two fields in same doc | Slower than regular queries |

---

## Key Takeaways

- Use **`$not`** and **`$nor`** for exclusion logic
- Use **`$regex`** for pattern matching — but always consider performance
- Use **`$expr`** when you need field-to-field comparison inside a document
- Combine operators for powerful, real-world backend filters
